What's New (v7 vs v6)

v6 — LLM-Rendered

The LLM generated both the formal structure (<step> types, from= references, <logic> fields) and the natural language.

  • 74% of steps had incorrect type labels
  • 13% of derived steps had missing from= references
  • Derived steps often didn't reference all required premises
v7 — Programmatic + LLM Text

The formal proof chain is built programmatically from the skeleton. The LLM only generates natural language text.

  • 100% correct step type labels
  • 100% correct from= references
  • Natural, varied text across diverse domains

How the Dataset Is Generated

1
Skeleton Library
270K abstract logical problems with facts, rules, proof steps, and answers across propositional, FOL, and non-monotonic logic.
2
Chain Builder
Programmatically constructs correct <step> XML with types, from-refs, and logic fields. Guaranteed correct.
3
LLM Text Gen
gpt-oss-120b generates entity/predicate maps, context paragraph, question, step descriptions, and inter-step reasoning.
4

6-stage check: JSON parse, mapping completeness, identifier validity, text quality, reasoning coverage, context grounding.

Step 1: Skeleton Library

Each skeleton defines an abstract logical problem:

Facts:         clear_headed(indie), well_prepared(walter)
Rules:         logical(X) :- clear_headed(X)
               intuitive(X) :- logical(X)
Disjunctions:  brave(indie) v cautious(indie)
Default rules: flies(X) :- bird(X), not ab_flies(X)
Proof steps:   Complete derivation chain with types and from-refs
Answer:        True / False / Unknown

Step 2: Programmatic Chain Builder

build_programmatic_chain() converts each skeleton's proof steps into correct XML. The chain builder guarantees:

  • Correct type="premise" vs type="derived" labels
  • Correct from= references (which premises each derived step depends on)
  • Correct <logic> fields with entity/predicate name mapping applied
  • Proper handling of Unknown answers (backward-chaining with unreachable()/unknown() meta-predicates)

Step 3: LLM Text Generation

The LLM receives the skeleton structure and produces:

  • entity_map — Maps abstract names to domain-appropriate names (e.g., alice → coral_reef)
  • predicate_map — Maps abstract predicates to domain properties (e.g., happy → thriving)
  • context — A natural language paragraph that explicitly grounds all premises
  • question — The query in natural language
  • step_texts — Natural language descriptions for each step
  • inter_step_reasoning — Explanatory text between steps

Step 4: Assembly & Validation

The programmatic chain and LLM text are combined into a complete completion. Multi-stage validation checks:

  1. JSON parsing (with truncation repair for long responses)
  2. Mapping completeness (all entities/predicates in proof chain are mapped)
  3. Identifier validity (mapped names are valid Prolog identifiers)
  4. Context/question quality (minimum length)
  5. Inter-step reasoning coverage
  6. Context grounding (mapped predicates and entities appear in context text)

Key Design Points

  • Structural correctness is guaranteed, not probabilistic. The formal proof chain (step types, from-references, logic expressions) comes from programmatic construction, not LLM generation. This eliminates the mislabeling and missing-reference problems of v6.
  • Three-class balanced. True/False/Unknown are evenly distributed (~33% each), which is critical for training models that don't collapse to a majority class.
  • Context grounds all premises. Every fact, rule, and disjunction in the formal proof appears as an explicit statement in the natural language context paragraph. Validation enforces this.
  • Unknown reasoning uses meta-predicates. For Unknown answers, the proof trace uses backward-chaining with unreachable(P) (P cannot be derived) and unknown(P) (neither P nor ¬P derivable), verified by both SAT and ASP solvers.
  • 20+ inference rules. Modus ponens, modus tollens, disjunctive syllogism, hypothetical syllogism, constructive/destructive dilemma, material implication, commutation, default reasoning, reasoning about exceptions, and priority-based reasoning.
  • Domain variety. The LLM maps each skeleton to a randomly assigned domain (marine biology, ancient history, nutrition, aviation, etc.), producing diverse and natural-sounding text.
